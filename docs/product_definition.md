## 1. 产品概述

### 1.1 产品名称
Camera Driver Agent (CDA) — 手机 Camera 驱动工程师智能助手

### 1.2 产品定位
多平台 Camera 驱动问题诊断与解决的 AI Agent，支持高通/MTK/展锐三大平台，按子平台和项目分级管理知识库，能够像资深 Camera 驱动工程师一样分析日志、审查代码、诊断问题并给出解决方案。

### 1.3 目标用户
| 用户角色 | 使用场景 | 核心诉求 |
|---------|---------|---------|
| Camera 驱动工程师 | 日常问题排查、代码审查 | 快速定位问题根因，获取解决方案 |
| BSP 工程师 | Camera 模组点亮 | 按步骤引导点亮流程，检查配置正确性 |
| 测试工程师 | Bug 初步定位 | 从日志中提取关键错误信息，判断问题归属 |

### 1.4 核心价值
- **提效**: 将驱动问题定位时间从小时级缩短到分钟级
- **降门槛**: 让初级工程师也能处理复杂的 Camera 驱动问题
- **知识沉淀**: 将资深工程师的经验固化为可复用的知识库
- **知识隔离**: 不同子平台知识库隔离，避免跨平台知识污染

## 2. 平台与项目管理

### 2.1 三级选择体系

用户使用 Agent 前需选择平台和项目，形成三级选择：

```
一级: 平台厂商 (Platform Vendor)
  ├── 高通 (Qualcomm)
  ├── MTK (MediaTek)
  └── 展锐 (UNISOC)

二级: 子平台 (Sub-Platform)
  ├── 高通 → SM8550 / SM8650 / QCM4490 ...
  ├── MTK  → MT6985(24E) / MT6989 / MT6897 ...
  └── 展锐 → T820 / T770 / T750 ...

三级: 项目 (Project)
  ├── MT6985 → 项目A / 项目B / 项目C ...
  ├── SM8550 → 项目X / 项目Y ...
  └── T820   → 项目M / 项目N ...
```

### 2.2 知识库隔离规则

| 维度 | 规则 | 说明 |
|------|------|------|
| 不同一级平台 | 完全隔离 | 高通/MTK/展锐的知识库完全独立，不共享 |
| 不同二级子平台 | 完全隔离 | 如 MT6985 和 MT6989 的知识库独立，不共享 |
| 相同子平台不同项目 | 共享平台知识库 | 项目A 和 项目B 共享 MT6985 的平台知识 |
| 项目级知识 | 项目私有 | 每个项目可以有项目特有的配置、经验等私有知识 |

## 3. 功能定义

### 3.1 核心功能

#### F1: 对话式问题诊断
- **描述**: 用户通过自然语言描述 Camera 驱动问题，Agent 通过多轮对话逐步定位根因
- **输入**: 自然语言问题描述
- **输出**: 问题诊断报告 + 修复建议
- **优先级**: P0

#### F2: 内核日志分析
- **描述**: 用户上传/粘贴 dmesg、kernel log、logcat 日志，Agent 自动提取 Camera 相关错误并分析
- **输入**: 日志文本 / 日志文件
- **输出**: 错误摘要 + 根因分析 + 修复建议
- **优先级**: P0

#### F3: 设备树(DTS)审查
- **描述**: 用户上传 DTS 文件，Agent 检查 Camera 节点配置的完整性和正确性
- **输入**: DTS 文件 / DTS 代码片段
- **输出**: 配置审查报告 + 修改建议
- **优先级**: P0

#### F4: MIPI Timing 参数校验
- **描述**: 校验 MIPI CSI 的 timing 参数是否符合规范，检查 HS-SETTLE、LP 时间等
- **输入**: Timing 参数配置 / Sensor datasheet 中的 timing 要求
- **输出**: 参数校验结果 + 修正建议
- **优先级**: P1

#### F5: 代码片段分析
- **描述**: 用户粘贴 Sensor Driver 代码片段，Agent 分析潜在问题
- **输入**: C/H 代码片段
- **输出**: 代码审查结果 + 修改建议
- **优先级**: P1

#### F6: 联网搜索
- **描述**: 当知识库无法覆盖时，Agent 联网搜索最新解决方案
- **输入**: 搜索关键词（Agent 自动生成）
- **输出**: 搜索结果摘要 + 相关解决方案
- **优先级**: P1

#### F7: 经验积累
- **描述**: 将成功的诊断-解决案例存入经验库，后续相似问题可直接引用
- **输入**: 对话历史（自动提取）
- **输出**: 经验条目
- **优先级**: P2

#### F8: 平台/项目管理
- **描述**: 管理平台、子平台和项目的增删改查，以及知识库的维护
- **输入**: 管理操作指令
- **输出**: 操作结果
- **优先级**: P0

### 3.2 功能边界 (不做)
- 不直接修改用户代码或设备树文件
- 不执行远程命令（不 SSH 到设备）
- 不替代正式的代码 Review 流程
- 不处理非 Camera 子系统的驱动问题（首期）

## 4. 知识域定义

### 4.1 预置知识库

#### 高通平台通用知识
| 知识类别 | 内容 | 来源 |
|---------|------|------|
| Camera 架构 | CAMSS/ISP 架构说明 | Qualcomm 官方文档 |
| Sensor 点亮指南 | 上电时序、I2C 配置 | 内部经验整理 |
| DTS 配置规范 | Camera DTS 节点属性说明 | Qualcomm 官方文档 |

#### MTK 平台通用知识
| 知识类别 | 内容 | 来源 |
|---------|------|------|
| Camera 架构 | ISP pipeline、3A、HAL3 架构说明 | MTK 官方文档 |
| Sensor 点亮指南 | 上电时序、I2C 配置、DTS 模板 | 内部经验整理 |
| MIPI CSI 规范 | Timing 参数定义、计算方法 | MIPI Alliance 规范 |
| 常见问题库 | 100+ 常见 Camera 驱动问题及解决方案 | 历史问题积累 |
| DTS 配置规范 | Camera DTS 节点属性说明 | MTK 官方文档 |
| 错误码速查 | Camera 相关 kernel 错误码含义 | Kernel 源码 |

#### 展锐平台通用知识
| 知识类别 | 内容 | 来源 |
|---------|------|------|
| Camera 架构 | 展锐 Camera 驱动架构说明 | UNISOC 官方文档 |
| Sensor 点亮指南 | 上电时序、I2C 配置 | 内部经验整理 |

### 4.2 知识更新机制
- 预置知识随版本发布更新
- 联网搜索结果不自动入库，需人工审核后纳入
- 经验库自动积累，定期清理低质量条目
- 项目级知识由项目成员维护，不影响其他项目
- 平台级知识更新后，该子平台下所有项目自动可见

## 5. 交互设计

### 5.1 交互方式
- **对话**: 自然语言问答，支持多轮对话
- **文件上传**: 支持上传 .log / .dts / .dtsi / .c / .h 文件
- **代码粘贴**: 支持直接粘贴代码片段
- **平台选择**: 启动时选择平台/子平台/项目

### 5.2 典型对话流程

```
用户: "我的 MTK 24E 平台上 imx586 sensor 点不亮，dmesg 报 i2c transfer failed"

Agent:
  1. [上下文] 当前平台: MTK / MT6985(24E) / 项目A
  2. [调用日志分析工具] 解析 i2c transfer failed 错误
  3. [调用知识库检索] 搜索 MT6985 Camera I2C 通信问题
  4. [推理] 可能原因: I2C地址错误 / 上电时序 / GPIO配置
  5. [追问] "请提供以下信息帮助定位:
     - Sensor 的 I2C 地址配置 (7位地址)
     - DTS 中 camera 节点的 regulator 配置
     - 完整的 dmesg log"

用户: [上传 dmesg.log 和 dts 文件]

Agent:
  1. [调用日志分析工具] 提取 Camera 相关日志
  2. [调用DTS解析工具] 检查 Camera 节点配置
  3. [推理] 发现 DTS 中 I2C 地址为 0x1a，但 imx586 应为 0x10
  4. [输出诊断报告]

诊断结果:
- 平台: MTK MT6985 (24E)
- 项目: 项目A
- 根因: I2C 地址配置错误
- 当前值: 0x1a
- 正确值: 0x10 (IMX586 SADDR 引脚接地)
- 修复方法: 修改 DTS 中 reg = <0x1a> 为 reg = <0x10>
- 参考文档: [MT6985 Camera Bring-up Guide]
```

### 5.3 输出格式
所有诊断结果以结构化报告输出：
- **当前平台**: 平台/子平台/项目
- **问题摘要**: 一句话描述问题
- **根因分析**: 详细分析问题根因
- **修复建议**: 具体的修改步骤/代码
- **参考链接**: 相关文档/社区讨论
- **置信度**: 高/中/低

## 6. 非功能需求

| 维度 | 要求 |
|------|------|
| 响应时间 | 单次对话响应 < 30s (不含联网搜索) |
| 准确率 | 诊断准确率 > 70% (首期目标) |
| 并发 | 支持 10 个并发用户 |
| 可用性 | 99% (开发阶段) |
| 数据安全 | 不存储用户上传的源码文件，对话记录加密存储 |
| 知识隔离 | 不同子平台知识库严格隔离，项目级知识项目私有 |
| LLM 可配置 | 支持用户自定义 LLM 提供商和模型，默认 MiniMax |
